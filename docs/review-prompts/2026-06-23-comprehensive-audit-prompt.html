15 監査観点（独立・並列）

1.コード品質（Swift / SwiftUI idiom）

  • force-unwrap (!) / force-try (try!) の濫用
  • @State / @StateObject / @Binding / @ObservedObject の選択ミス
  • onAppear での副作用の重複発火
  • Task { } / async/await のキャンセル漏れ
  • 不要な @MainActor 指定 / 漏れ
  • Long view body (200 行超 / 5 階層超のネスト)

2.アーキテクチャ / 依存関係

  • Service 層と View 層の責務分離 (Services/ 配下の design)
  • Singleton 化された UserDefaults アクセスの testability
  • Navigation Stack の path 管理が view ごとに散らばっていないか
  • Widget Extension と App の data sharing 戦略（App Group の必要性）
  • 循環参照（retain cycle）

3.パフォーマンス / 起動時間

  • 1191 件 games.json の初回読込み戦略（同期 / 非同期 / lazy）
  • ScenePicker / Proposal シャッフルの計算量
  • Image / SF Symbol の重い render
  • SwiftUI の不要な再描画（ID 不整合・equatable 漏れ）
  • cold start > 400ms になっていないか

4.アクセシビリティ

  • VoiceOver: 各画面の読み上げ順序が論理的か
  • Dynamic Type: xLarge / accessibility5 で破綻しないか
  • 色のみで情報を伝えていないか（color-blind safe）
  • contrast ratio 4.5:1 以上（docs/color-audit.html 参照）
  • Reduce Motion 対応の完全性
  • Touch target 44×44pt 未満の箇所

5.ローカリゼーション / 国際化

  • xcstrings の en 翻訳の品質（不自然・直訳・文化適合）
  • ハードコード文字列の取り残し（Text("...") 直書きで xcstrings 未登録）
  • 右から左の言語（アラビア・ヘブライ）への対応有無
  • regionBlocklist 6 件 × イスラム圏指定が App Store の availability 設定と整合しているか
  • 日付・数値・通貨のロケール対応

6.App Store 審査リスク（Guidelines 違反）

  • 飲酒・罰ゲーム表現が 12+ レーティングと矛盾しないか
  • 他社製品名（GeoGuessr / Among Us / Gartic Phone / Jackbox / Codenames / Skribbl.io）が 実データに残存している remote シーンが商標リジェクト対象にならないか
  • 「アイデア箱」が mailto: で外部メーラを開く動作が UX guideline 違反でないか
  • Privacy Manifest の Required Reason API 宣言漏れ
  • 4.5 (Spam / Repackaged) 対象になる「シーン提案」コアの独自性をどう主張するか
  • Age rating questionnaire の回答との整合性

7.セキュリティ / プライバシー

  • satori7227@gmail.com を SuggestionFormView.swift にハードコードしている件のリスク
  • UserDefaults に保存される個人データ（お気に入り / コレクション / フィードバック）の機微度
  • iCloud バックアップに乗るデータの量と機微度
  • Universal Link や Custom URL scheme の悪用可能性（asobi://）
  • 本人確認なしで開ける「設定」「アイデア箱」が悪意ユーザーから乱用されるシナリオ

8.データ整合性 / コンテンツ品質

  • 1191 件 games.json の schema が全件均質か（必須キー欠落、型不整合）
  • 各 game の scenes 配列が空 / 未定義シーン参照を含んでいないか
  • シーン別ゲーム数の偏り（drinking 574 vs konkatsu 101 等）が UX 上問題か
  • 同一名 / 同一説明のゲーム重複
  • nameEn / summaryEn / rulesEn が 80 件のみで他は日本語 → 海外利用時の体験
  • regionBlocklist の漏れ（飲酒系の他のゲームに blocklist が付いていない可能性）
  • prompts.json カテゴリ間の難易度バランス

9.テスト品質 / 回帰耐性

  • 88 unit tests のカバレッジ偏り（Services 層集中 vs View 層空白）
  • games.json 変更時に検知できる schema test の有無
  • UI test 1 件だけでは Phase β / γ の IAP / サブスク追加時にカバー不足
  • snapshot test の不在（UI regression 検知能力）
  • fastlane test が CI に組み込まれていない（手動実行のみ）

10.リリースプロセス / CI/CD

  • GitHub Actions / Xcode Cloud が未設定 → 手動 fastlane に依存
  • fastlane API Key が ~/.config/asobi/.envrc に置く想定で安全か（誤コミットリスク）
  • version bump （Version.xcconfig）の自動化と人手介入の境界
  • Phased Release 中断時のリカバリ手順の docs 化
  • asobi-app と asobi-privacy の 2 リポ間の同期手順（privacy 内容更新時）

11.収益化準備 / IAP / StoreKit2

  • PurchaseStore.swift がまだ UserDefaults だけで実 StoreKit2 接続なし → Phase β 着手時の差分量
  • Constants.softPaywallEnabled / premiumEnabled / tipJarEnabled の解禁順序設計
  • Family Sharing と Subscription Group の組み立て案
  • Receipt validation 方針（オンデバイス vs サーバー）
  • サブスク 7 日間無料トライアルの設計
  • Small Business Program 適格性の維持条件

12.マーケティング / ASO

  • サブタイトル「1191ゲームを場面別に提案。手ぶらで遊べる」の検索ヒット率と訴求
  • キーワード 93 字の配分（飲み会・パーティー系 vs シーン特化）
  • スクショ 5 枚の物語性（最初の 3 枚で離脱止められるか）
  • Promotional Text の月次差し替え運用（季節文脈）
  • note / X / LinkedIn 告知タイミングと文面（feedback_publish_includes_announcement 準拠）
  • Featuring Nomination の応募タイミング

13.法務 / 規約

  • プライバシーポリシー（satori7227-dot.github.io/asobi-privacy/）が最新コードと一致しているか
  • 利用規約（EULA）を別途置くか Apple の標準を流用するか
  • 商標：「ASOBI」「あそぶ？」の商標調査・出願要否
  • games.json の各ゲームに既存の商標がある場合の表記（「山手線ゲーム」「ロシアンルーレット」等）
  • 送別会 / 同窓会など職場文脈の利用での雇用主データ取扱い責任
  • 個人開発者の特定商取引法表示（IAP 解禁時に必要）

14.ドキュメント整合性 / 真実の単一情報源

  • README.md / launch-prep / post-developer-signup / release-checklist 間で数字や手順が矛盾していないか
  • 13 シーン名の表記揺れ（「カップル」vs「カップル・2 人」等）
  • implementation-notes と現状コードの乖離（古い決定が残っていないか）
  • scripts/ ヘルパ群が docs から正しく参照されているか
  • 図やテーブルが現状を反映しているか（256 ゲーム → 1191 ゲーム の伝播漏れ）

15.運用 / 1 人開発の継続可能性

  • Apple ID / GitHub PAT / fastlane API Key / 銀行口座情報など機密の集約場所
  • 「30 日休んだら何が壊れるか」リスト（fastlane key 期限・PAT 期限・サブスク期限・証明書期限）
  • 本人不在時のユーザーサポート（意見箱への返信遅延）
  • iOS メジャーアップデート（17 → 18 → 19 → 26）への追従コスト
  • Anthropic 月 $25 縛りの中で AI 機能（v2 ヘルメスエージェント）を載せる方法

【severity 基準】

<h2>使い方</h2>

<h3>A. Claude Code の /code-review ultra で回す（最大規模・並列）</h3>
<pre>cd /Users/idogawasatori/Desktop/AppDev/asobi-app
# 現在のブランチで multi-agent cloud review を起動
/code-review ultra

# プロンプトの指示文として上記をコピペ

<h3>B. Workflow で自前 panel を組む（細かい制御）</h3>
<pre># 15 観点それぞれを独立 agent として fan-out
# 適応度関数: severity 配分（Critical/High に集中させる）
# 統合: 全 findings を dedup → 優先順位付き ToDo 生成

claude --workflow review-panel \
  --args '{"scope": "asobi-app", "lenses": 15, "depth": "high"}'</pre>

<h3>C. 単発 Agent で回す（軽量）</h3>
<pre># Agent ツールに上記プロンプトをそのまま渡す
# subagent_type=general-purpose または code-reviewer
# 一度に 15 観点を 1 agent でやらせる場合は出力が長大化するため、
# 4-5 観点ずつ 3 セッションに分けるのが現実的</pre>

<h3>D. 他社 LLM （Gemini / GPT / o3）で回す（コスト分散）</h3>
<pre># プロンプト本体（pre 内）をそのまま貼り付け
# 各 LLM の長文出力上限に注意（≦ 32k tokens 推奨）
# 同じプロンプトで複数 LLM の結果を取って比較すると、
# 1 LLM では見逃される盲点が出やすい</pre>
